# 需求澄清与AC定义

> 本文档基于已实现基础的订单创建和查询功能的系统，因此假设：**用户已经正常登录且其他订单项都满足对应条件**

---

## 需求分析

1. **折扣计算**：基于订单的**商品总价（itemsTotal）**判断是否达到满减门槛（**不含**包装费、配送费）
2. **满减规则定义**：
   - 门槛金额（threshold）：触发优惠的最低商品总价
   - 优惠金额（discountAmount）：满足条件时减免的金额
   - 规则由商家预先配置，系统读取使用
3. **折扣应用规则**：
   - 当商品总价 **≥** 门槛金额时，最终金额 = 原最终金额 - 优惠金额
   - 优惠后金额最低为0元（不会出现负数）
4. **多规则策略**：当订单同时满足多个满减规则时，选择**优惠金额最大**的规则应用
5. **无规则/无效规则**：商家未配置规则或规则无效时（门槛<0或优惠金额≤0），不应用任何优惠
6. 订单响应中应包含折扣信息

7. **不变性原则**：订单创建完成后，后续规则变更不影响已存在的订单
8. **扩展性**：设计需支持未来新增其他折扣类型（如折扣券、会员价）

## 边界情况

1. 订单商品总价 **正好等于** 门槛（如50元）——应触发优惠（≥门槛）
2. 订单商品总价 **略低于** 门槛（如49.99元）——不触发优惠
3. 订单商品总价 **远高于** 门槛（如100元）——只触发一次优惠（非阶梯/叠加）
4. 商家 **未配置** 任何满减规则——订单无优惠
5. 规则配置 **无效**（门槛<0 或 优惠金额≤0）——视为无优惠
6. 订单满足 **多个规则** 的条件——选择优惠金额最大的规则应用
7. 优惠后最终金额 **可能为负数**——最终金额最低为0元
8. 订单商品总价为 **0或负数**（极端情况）——不触发优惠
9. **金额精度**：所有计算保留两位小数，遵循四舍五入

10. **规则变更**：订单创建后规则修改，不影响已创建的订单

## 验收标准

1. **基础功能验证**
   - [ ] 用户下单时，若商品总价 ≥ 门槛，最终金额正确减去优惠金额
   - [ ] 用户下单时，若商品总价 < 门槛，最终金额不变
   - [ ] 边界金额（商品总价 = 门槛）能正确触发优惠
2. **多规则验证**
   - [ ] 商家配置多个规则时，订单选择优惠金额最大的规则应用
   - [ ] 验证选择正确性（如：满50减5、满100减15，100元订单应减15）
3. **无规则/无效规则验证**
   - [ ] 商家未配置规则时，订单价格计算正常，无优惠
   - [ ] 规则无效（门槛<0、优惠金额≤0）时，订单无优惠
4. **响应格式验证**
   - [ ] 订单创建/查询响应中包含折扣信息：

5. **业务规则验证**
   - [ ] 优惠后金额不低于0元（如满5减10，订单应付0元）
   - [ ] 金额计算精度正确（两位小数，四舍五入）
   - [ ] 订单创建后，规则修改不影响已存在订单

6. **扩展性验证**
   - [ ] 现有测试用例在新增折扣类型后依然通过

7. **测试覆盖要求**
   - [ ] 单元测试覆盖所有边界情况
   - [ ] 集成测试验证API响应格式
   - [ ] 整体测试覆盖率 ≥ 70%（符合项目要求）
